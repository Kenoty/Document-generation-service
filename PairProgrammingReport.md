# Впечатления о парном программировании при создании проекта `Document-generation-service`

## 1. Какие задачи были реализованы:

---

### Задача 1: Система аутентификации и управление пользователями

**Что делали:**

- Реализация `AuthController` с эндпоинтами для регистрации, входа, выхода
- Настройка сессионной аутентификации через `HttpSession`
- Создание модели `User` и `UserRepository`
- Реализация `UserService` для бизнес-логики работы с пользователями
- Настройка `SecurityConfig` для защиты эндпоинтов

---

### Задача 2: Работа с шаблонами документов

**Что делали:**

- Реализация `TemplateController` с CRUD-операциями для шаблонов
- Создание модели `Template` и `TemplateRepository`
- Реализация `TemplateService` для работы с шаблонами
- Добавление загрузки DOCX-файлов через `FileProcessingService`
- Извлечение полей-плейсхолдеров из содержимого шаблона
- Создание `TemplateDTO` для передачи данных на фронтенд

---

### Задача 3: Генерация и экспорт документов

**Что делали:**

- Реализация `DocumentController` для генерации документов
- Создание модели `Document` и `DocumentRepository`
- Реализация `DocumentService` для бизнес-логики
- Добавление экспорта в HTML, DOCX, PDF через `FileProcessingService`
- Создание `InMemoryMultipartFile` для работы с файлами в памяти
- Привязка документов к конкретному пользователю

---

### Задача 4: Пакетная генерация документов

**Что делали:**

- Реализация `BatchDocumentService` для обработки нескольких документов за раз
- Создание DTO для пакетной обработки (`BatchGenerationRequest`, `BatchGenerationResult`, `BatchItemResult`, `BatchError`)
- Поддержка загрузки данных в формате CSV/Excel
- Генерация ZIP-архива с результатами
- Обработка ошибок для каждого документа отдельно

---

### Задача 5: Гостевой доступ (новая функциональность)

**Что делали:**

- Реализация rate limiting (ограничение количества запросов)
- Ограничение размера загружаемых файлов (10MB)
- Временное хранение документов с истечением через 24 часа
- Добавление опции сохранения гостевых документов при регистрации

---

### Задача 6: Фронтенд на React

**Что делали:**

- Создание компонентов для отображения списка шаблонов и документов
- Реализация форм для создания шаблонов и генерации документов
- Добавление интерфейса для гостевого доступа
- Интеграция с бэкенд-эндпоинтами через API-клиент

---

## 2. Какие стили парного программирования попробовали

Мы работали втроем и распределялись по парам в зависимости от задачи. Каждый участник поработал с каждым в разных ролях.

### Стиль 1: Ведущий-ведомый

**Где использовали:**

- Настройка `SecurityConfig` и сессионной аутентификации
- Реализация `FileProcessingService` для работы с DOCX и PDF
- Настройка rate limiting для гостевого доступа

**Как работали:**

- Более опытный в конкретной технологии объяснял подход
- Менее опытный писал код под руководством
- Ведущий проверял каждый шаг и объяснял, почему делаем так, а не иначе

**Пример:** При настройке безопасности один участник хорошо знал Spring Security, двое других только начинали с ним работать. Ведущий объяснил, как работают фильтры, сессии, почему важно настраивать CORS именно так.

---

### Стиль 2: На равных

**Где использовали:**

- Разработка `BatchDocumentService` и пакетной обработки
- Создание DTO и маппинг данных
- Рефакторинг контроллеров

**Как работали:**

- Два разработчика с примерно одинаковым опытом
- Постоянно менялись ролями (каждые 30-40 минут)
- Обсуждали каждое решение, находили оптимальный подход

**Пример:** При разработке пакетной генерации обсуждали, как лучше организовать обработку ошибок — прерывать весь пакет или продолжать и собирать отчет. Приняли решение продолжать и возвращать список ошибок.

---

### Стиль 3: Водитель-штурман

**Где использовали:**

- Проектирование архитектуры гостевого доступа
- Разработка системы временного хранения документов
- Интеграция всех компонентов в единое целое

**Как работали:**

- Один писал код (водитель)
- Второй смотрел на общую картину, предлагал улучшения, искал узкие места (штурман)

**Пример:** При проектировании гостевого режима штурман заметил, что нужно не только ограничивать количество запросов, но и чистить старые записи, чтобы не забивать память. Добавили `@Scheduled` задачу для очистки.

---

### Стиль 4: Пинг-понг (частично)

**Где использовали:**

- Тестирование критичных методов (хотя полноценных тестов у нас нет, использовали ручное тестирование по принципу пинг-понга)

**Как работали:**

- Один писал сценарий ручного теста
- Второй проверял, работает ли код по сценарию
- Менялись ролями

**Пример:** При проверке rate limiting один участник писал скрипт для быстрой отправки 10 запросов подряд, второй смотрел, срабатывает ли ограничение после 5 запросов.

---

## 3. Какой стиль показался более приемлемым? Почему?

### Самым эффективным оказался **смешанный подход** — выбор стиля под конкретную задачу


| Стиль                | Когда использовали                               | Эффективность            |
| -------------------- | ------------------------------------------------ | ------------------------ |
| **Ведущий-ведомый**  | Сложные темы (безопасность, работа с файлами)    | Лучший для обучения      |
| **На равных**        | Задачи средней сложности, где мнения равнозначны | Быстрое принятие решений |
| **Водитель-штурман** | Проектирование архитектуры, новые модули         | Меньше ошибок в дизайне  |
| **Пинг-понг**        | Тестирование, отладка                            | Полезно, но медленно     |


### Почему именно так:

1. **Разнообразие задач** — в проекте есть и сложные архитектурные решения (гостевой режим), и рутинные операции (CRUD для шаблонов). Под каждый тип задач нужен свой стиль.
2. **Разный уровень участников** — кто-то сильнее в бэкенде, кто-то в React. Ведущий-ведомый позволяет передавать знания.
3. **Сложность проекта** — генерация документов, работа с файлами, безопасность — темы, где легко ошибиться. Штурман помогает не упустить детали.
4. **Время** — на равных работаем быстрее, когда оба понимают задачу. Ведущий-ведомый медленнее, но качественнее.

## 4. Что получилось? Положительные впечатления

### Качество кода

Код, написанный в паре, требовал меньше исправлений. Особенно это заметно на `SecurityConfig` и `FileProcessingService` — сложные места, где легко ошибиться.

### Обмен знаниями

- Один участник научился работать с `MultipartFile` и загрузкой файлов
- Второй разобрался с сессионной аутентификацией и `HttpSession`
- Третий прокачал навыки работы с React и интеграцией с API

### Быстрое решение проблем

Когда сталкивались с ошибками (например, CORS при запросах с фронтенда), в паре находили решение в 2-3 раза быстрее, чем поодиночке.

### Меньше багов

При разработке гостевого режима в стиле "водитель-штурман" сразу заметили, что нужно чистить старые сессии — добавили планировщик. Если бы писали по одному, могли забыть.

### Дисциплина

В паре меньше хотелось отвлекаться на телефон или соцсети. Работа шла более сфокусированно.

### Понимание всего проекта

Раньше каждый отвечал за свою часть (один — за аутентификацию, другой — за шаблоны). После парного программирования все трое понимают, как устроен весь проект.

## 5. Что не получилось? Как можно это поправить?

### Проблема 1: Усталость при долгой работе

**Что случилось:** После 3-4 часов парного программирования концентрация падала, начинали делать глупые ошибки.

**Как поправить:**

- Ввели правило: не больше 2 часов подряд в паре, потом перерыв
- Чередуем пары (сегодня А+Б, завтра А+В, послезавтра Б+В)
- Используем Pomodoro (45 минут работы, 10 минут отдыха)

---

### Проблема 2: Не все задачи удобно делать в паре

**Что случилось:** Пытались писать простые DTO и репозитории в паре — было скучно и неэффективно.

**Как поправить:**

- Четко договорились, какие задачи делаем в паре, а какие — соло
- В паре делаем только сложные или критические задачи
- Рутинные задачи распределяем и делаем индивидуально

---

### Проблема 3: Споры о форматировании кода

**Что случилось:** Тратили время на обсуждение, ставить пробел или нет, использовать табы или пробелы.

**Как поправить:**

- Настроили единый форматтер (Prettier для React, Spring Java Format для бэкенда)
- Договорились не обсуждать форматирование — доверяем автоформатированию

---

### Проблема 4: Один участник доминировал

**Что случилось:** В некоторых задачах более опытный участник слишком сильно давил авторитетом, меньше слушал мнение других.

**Как поправить:**

- Ввели правило: перед тем как писать код, каждый высказывает свое мнение
- Используем таймер на смену ролей (чтобы все побывали и водителем, и штурманом)

---

### Проблема 5: Неудобно сидеть

**Что случилось:** При смене водителя приходилось пересаживаться или тянуться к клавиатуре.

**Как поправить:**

- Купили удлинитель для клавиатуры и мыши
- Используем VS Code Live Share для удаленной работы (даже сидя рядом)
- Ноутбук ставим на подставку, чтобы всем было видно

---

## 6. Будете ли использовать ПП для других задач? Каких?

**Да, планируем использовать парное программирование для:**

### Сложные и критичные задачи

- Добавление ролевой модели (админ, пользователь, гость) — сейчас у нас только базовая аутентификация
- Реализация очередей для пакетной генерации (сейчас все синхронно)
- Интеграция с внешними сервисами (например, отправка документов по email)
- Доработка безопасности (защита от CSRF, настройка CORS)

### Архитектурные решения

- Проектирование новых модулей (например, версионность шаблонов)
- Рефакторинг существующего кода (выделение общих сервисов)

### Отладка сложных багов

- Когда ошибка не очевидна, свежий взгляд помогает быстрее найти причину
- Баги, связанные с многопоточностью или конкурентным доступом

### Онбординг новых участников

- Если в команду придет новый разработчик, будем использовать ведущий-ведомый для погружения в проект

### Написание тестов (когда добавим)

- Планируем в будущем добавить тесты — тогда будем использовать пинг-понг

### Где НЕ будем использовать:

- Создание DTO и репозиториев (рутина)
- Написание миграций БД
- Исправление опечаток и мелких багов
- Обновление зависимостей

### Плюсы парного программирования для нашего проекта:


| Аспект         | Результат                                                |
| -------------- | -------------------------------------------------------- |
| Качество кода  | Выросло, особенно в сложных местах (безопасность, файлы) |
| Скорость       | Чуть медленнее, но меньше переделок                      |
| Обмен знаниями | Все трое теперь понимают весь код                        |
| Командный дух  | Стали лучше коммуницировать                              |

| Дисциплина     | Выше, чем при solo-работе                                |


### Минусы:

- Требует удобного рабочего места
- Не подходит для простых задач
- Утомляет при долгой работе
- Нужна психологическая совместимость

### Итог:

Парное программирование — это **инструмент, который нужно применять точечно**. Для сложных задач (гостевой режим, безопасность, пакетная генерация) он незаменим. Для простых — только замедляет.
