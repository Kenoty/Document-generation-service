# Анализ технического долга проекта `Document-generation-service`

## 1. Введение

Целью данной работы является инспектирование проекта `Document-generation-service` на наличие признаков технического долга, разработка плана мероприятий по его устранению, оценка плана и сравнение объёма долга с объёмом нереализованных функций (фич). Анализ проведён на основе анализа кодовой базы проекта, доступной в репозитории.

---

## 2. Инспекция проекта на наличие технического долга

На основе предоставленного списка признаков и анализа структуры и содержимого проекта, выявлены следующие проявления технического долга:

| Признак технического долга | Наличие в проекте | Обоснование / Примеры |
| :--- | :--- | :--- |
| **Непонятный / нечитабельный код** | **Да** | В проекте отсутствуют линтеры и единые стандарты кодирования. В Java-части (`DocumentController.java`, `DocumentService.java`) некоторые методы превышают 40-50 строк, отсутствуют комментарии к сложным участкам (например, логика генерации документов). В React-части (`DocumentGenerator.js`) компонент содержит избыточную логику обработки состояния, не вынесенную в отдельные хуки. Методы и переменные имеют не всегда говорящие названия (например, `processData`, `handleStuff`). |
| **Дублирующийся код** | **Нет** | Внутри Java и React частей явного дублирования кода не обнаружено. Java-код разделён по слоям (контроллер, сервис, репозиторий), React-код — по компонентам и сервисам. |
| **Отсутствие автоматизации (тестов, сборки, развёртывания)** | **Да** | В проекте отсутствуют автоматические тесты. В Java-части нет модульных тестов (отсутствует директория `src/test/`). В React-части нет тестов компонентов. Отсутствуют конфигурации для CI/CD. Нет автоматизации сборки и развёртывания — процесс выполняется вручную. |
| **Запутанная архитектура и ненужные сложные зависимости** | **Да** | В контроллере (`DocumentController`) присутствует бизнес-логика, которая должна быть вынесена в сервисный слой. Отсутствует чёткое разделение на слои в React-части (например, не выделен слой для работы с API). |

| **Незакоммиченый код / долгоживущие ветки** | **Нет** | В проекте не обнаружено признаков долгоживущих веток или незакоммиченного кода. Разработка ведётся в основной ветке с регулярными коммитами. |
| **Отсутствие / несоответствие технической документации** | **Да** | `README.md` содержит только общее описание проекта. Отсутствует документация по API (эндпоинты, форматы запросов/ответов). Нет описания архитектуры проекта. В коде отсутствуют комментарии, поясняющие сложные участки логики. |
| **Отсутствие тестовой среды** | **Да** | В проекте нет конфигураций для развертывания тестовой среды: отсутствуют Dockerfile, docker-compose, инструкции по настройке окружения для тестирования. Нет выделенной тестовой базы данных. |

## 3. План мероприятий по устранению технического долга

План разбит на этапы, упорядоченные по приоритету и логической последовательности.

1.  **Рефакторинг архитектуры:**
    *   **Действие:** Вынести бизнес-логику из контроллеров в сервисный слой. Разделить ответственность в React-компонентах: вынести логику работы с данными и API-вызовы в отдельные сервисы или хуки.
    *   **Цель:** Улучшить поддерживаемость кода, следовать принципу единственной ответственности.

2.  **Внедрение стандартов кодирования:**
    *   **Действие:** Добавить конфигурации для линтеров (Checkstyle для Java, ESLint для React). Настроить форматирование кода. Провести рефакторинг существующего кода для приведения к единому стилю. Переименовать методы и переменные для улучшения читаемости.
    *   **Цель:** Повысить читаемость и поддерживаемость кода, снизить вероятность ошибок.

3.  **Автоматизация сборки и развертывания:**
    *   **Действие:** Создать `Dockerfile` для бэкенда и фронтенда, а также `docker-compose.yml` для поднятия всего стека локально. Настроить сборку проекта (Maven/Gradle для Java, npm для React).
    *   **Цель:** Обеспечить идентичность сред разработки, тестирования и продакшена, упростить развертывание.

4.  **Внедрение Continuous Integration (CI):**
    *   **Действие:** Настроить CI-пайплайн (GitHub Actions), который будет автоматически запускать сборку и линтеры при каждом пуше. После добавления тестов — включить их в пайплайн.
    *   **Цель:** Обеспечить раннее выявление проблем интеграции, гарантировать работоспособность кода.

5.  **Разработка стратегии тестирования и добавление тестов:**
    *   **Действие:** Добавить юнит-тесты для ключевых модулей бэкенда (JUnit для сервисов). Добавить тесты для критически важных React-компонентов (Jest, React Testing Library). Покрыть тестами основные сценарии генерации документов.
    *   **Цель:** Создать "страховочную сетку" для рефакторинга и разработки новых фич, снизить риск регрессии.

6.  **Создание тестовой среды:**
    *   **Действие:** Настроить тестовое окружение с использованием Docker. Добавить тестовую базу данных (например, H2 или Testcontainers).
    *   **Цель:** Обеспечить изолированное и воспроизводимое тестирование.

7.  **Актуализация документации:**
    *   **Действие:** Обновить `README.md`, добавив инструкции по запуску, сборке и тестированию. Добавить документацию по API (OpenAPI/Swagger) с описанием эндпоинтов, форматов запросов и ответов. Добавить комментарии к сложным участкам кода.
    *   **Цель:** Снизить порог входа для новых разработчиков, обеспечить понимание кода.

## 5. Сравнение объёма долга и недоимплементированных фич

*   **Объём технического долга:** Долг затрагивает архитектуру (смешение ответственности в контроллерах и компонентах), качество кода (отсутствие стандартов, длинные методы), процессы разработки (отсутствие CI, тестов, автоматизации) и документацию. Проект находится в активной разработке, объём долга значителен, но его исправление реалистично.
*   **Объём недоимплементированных фич:** На основе анализа кода можно сделать вывод, что реализован базовый функционал генерации документов. Отсутствуют: поддержка различных форматов, расширенная система шаблонов.

**Вывод:** Объём недоимплементированных фич несколько **превышает** объём технического долга. Однако текущий технический долг (особенно в архитектуре) существенно замедляет реализацию нового функционала. Без его устранения добавление новых возможностей будет приводить к накоплению ещё большего долга и росту технической энтропии.

## 6. Выводы и коррективы в план разработки

1.  **Необходимость устранения долга:** Устранение технического долга является необходимым условием для эффективного развития проекта. Игнорирование долга приведёт к замедлению разработки, увеличению количества багов и усложнению поддержки.

2.  **Приоритеты выполнения:**
    *   **Этап 1 (Рефакторинг архитектуры)** — выполнить в первую очередь, так как текущая архитектура затрудняет добавление нового функционала и тестирование.
    *   **Этапы 2, 3, 4 (Стандарты, Автоматизация, CI)** — следующий приоритет, создающий основу для стабильной разработки.
    *   **Этапы 5, 6 (Тесты и Тестовая среда)** — критически важны для обеспечения качества и должны выполняться параллельно с разработкой.
    *   **Этап 7 (Документация)** — обновлять по мере изменения кода.

3.  **Изменения в процессе разработки:**
    *   Приостановить разработку новых крупных фич до выполнения этапов 1-4.
    *   Внедрить практику код-ревью с проверкой соответствия стандартам.
    *   Ввести **Definition of Done (DoD)**:
        *   Код проходит линтер.
        *   Написаны юнит-тесты для новой логики.
        *   CI-пайплайн успешен.
    *   Выделять 20% времени каждого спринта на сокращение технического долга.

Реализация предложенных мероприятий позволит стабилизировать проект и создать основу для эффективной разработки нового функционала.
